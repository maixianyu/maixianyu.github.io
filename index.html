数据流与Codelet模型[1]

发表于 2017-11-11 | 分类于 众核

无论是服务器还是用户手持终端，处理器的核数呈持续增加的趋势。新闻的用语也从CPU，到单核双核，多核，众核。而对于一段程序与操作系统而言，在单核处理器的程序执行模型（PXM，Program Execution Model）并不适用于众核，最明显的原因莫过于数据的共享方式与计算资源的利用方式会发生巨大的变化。为了解决在众核时代计算机的程序执行模型问题，高光荣教授团队提出了基于数据流的Codelet模型，而数据流思想由J.B.Dennis教授在20世纪70年代提出。
Codelet模型的基本特征是细粒度（fine-grain），事件触发执行（event-driven）。细粒度是相对传统线程调度模型的粗粒度而言。粗粒度情况下，只有线程需要花较长时间等待如存储器数据读写时间，才会对线程进行调度，调度的单元是大段的程序，所以称为粗粒度。粗粒度的不足是需要切换上下文，移动许多代码与数据。另外，类似MPI、OpenMP的并行程序是由人为发出并行指令执行（a control-flow version of parallel programs），所以很难充分发挥异步的潜力。

为了发挥Codelet模型的潜力，高光荣团队设想了一个抽象的众核计算机群结构，如图1所示。通过互联构成多节点计算机群，每个节点由多个处理器与内存芯片组成，每个处理器芯片上由多个cluster互联组成，而每个cluster内部由多个处理单元CU、一个同步单元SU和一个本地内存组成。这个机器结构是对未来并行计算机结构的抽象。
在Codelet模型中，供系统调度的最小粒度成为codelet。一个codelet由一小段机器指令组成，是对于调度系统不能再细分的计算单元，也就是具有运算的“原子性”。一个codelet的代码往往比传统粗粒度包含的一大块代码要小得多，体现了模型的细粒度。因此，当一个codelet运行在计算单元时，计算过程是不容许被中断的，意味着codelet的运行包含了“非抢占”性质。模型的设计者认为，虽然粒度变小意味着上下文开销更大，但是相信配合底层的硬件设计，并行的优势能打过上下文开销。同时也认为，长延时的等待也比频繁多任务的切换具有更低成本（it may be more power efficient to clock-gate a processing element while waiting for long latency operations than to multitask）。

多个codelet之间既有平行又有顺序关系。如图2所示为Codelet Graph Model，每个圆圈代表一个codelet，箭头代表了codelet之间数据的依赖与传递关系。依赖关系包含了多种事件，如内存的读取，中断的信号等等。数据或信号没到来时，codelet处于dormant状态；事件信号齐全时，处于enabled状态；开始执行时，称为fired。而如果多个codelet是在同一个方框里，说明它们共享了本地的数据，而每一个方框称为Threaded Procedure。Codelet和TP是模型的基本单元。执行模型里的Codelet和TP分别对应了抽象机器的CU和cluster。
模型设计者还强调，现有的并行编程语言Concurrent Collection,X10,Chapel都能使用Codelet执行模型，因为Codelet可以作为高级编程语言与机器语言的中间层，只要存在codelet-aware complier，就能翻译成Codelet模型运行。

参考文献：
[1] Zuckerman S, Suetterlein J, Knauerhase R, et al. Position paper: Using a codelet program execution model for exascale machines[C]//EXADAPT Workshop. 2011.

TOP500超级计算机的变化趋势

发表于 2017-10-06 | 分类于 计算机

TOP500项目是针对全球已知最强大的电脑系统做出排名与详细介绍。此项目始于1993年并且一年出版两次最新的超级计算机排名列表。每年的第一次排名公布总是在六月份的国际超级计算机会议上，而第二次排名公布则是在十一月份的超级计算会议上。此项目主旨在提供一个可靠的基础去追踪与侦测高性能计算的趋势[1]。

TOP500的排名只由一个计算机性能指标决定，即每秒进行浮点运算的次数，一般用 GFLOPS 表示。具体的计算方法为[2]：

GFLOPS = #chassis #nodes/chassis #sockets/node #cores/socket GHz/core * FLOPs/cycle

说到推动计算机技术快速发展的因素，就不能不提摩尔定律。摩尔定律最初的意思是积体电路上可容纳的电晶体（晶体管）数目，约每隔两年便会增加一倍[3]。有人从中引申出“预计18个月会将芯片的性能提高一倍”。在2017年的今天，摩尔定律也走到了10nm这一步，但是计算机性能的提升有多大程度上依然依赖摩尔定律，也许不是一句引申能定夺的。因此，笔者从TOP500[1]简单整理了过去10来年的数据，总结过去超算发展的规律，以及预测未来超算的变化趋势。

单从性能指标上看，超算领域在过去十年的计算性能维持着指数型的增长。中国的神威·太湖之光以93,014.6 TFLOPS/s的性能从2016年6月开始就一直稳居TOP500榜首，研发团队是国家并行计算机工程技术研究中心。

Python WiFi数据采集性能优化

发表于 2017-09-29 | 分类于 Python

背景

wxPython 利用 Twisted 网络模块接收采集模块通过 WiFi 传送的数据。接收之后按照协议解析、显示、存储。采集端因为没有精确的时钟，因此难以保证稳定的采集频率，例如 900 Hz 的采集设置并不能保证采集端每秒传送 900 组数据到 PC，并且由于网络环境的变化，PC 接收到的数据有时候每秒 1000 组，有时候每秒 700 组，因此，精确稳定的采集需求没有办法保证。
在这个背景下，我们提出由 PC 端保证采集频率的方法。同时，通过添加多个线程来保证数据的存储与显示流畅进行。

模型设想

精确采集的模型

为了完成精确采集，发送端与接收端共同建立如下模型：

  1. 发送端数据传送速率可以有波动，但是最低值须高于采集设置值，避免信息丢失。
  2. 接收端维持一个临时的队列，存放已接受但尚未处理的数据。
  3. 接收端通过定时器从队列中处理 1 秒钟的数据量，实现精确采集。

第 2 条引发的子问题是，这个队列的长度如何维持，如果对待传送过快而出现的冗余数据？笔者的方法是：

  1. 队列的长度为 2 秒钟的数据量长度。多出的 1 秒能够在传送速度偏慢时提供稳定的数据流，产生持续精确的“错觉”。
  2. 当传送数据过慢，队列长度小于 1 秒数据量时，暂停定时器与数据解析存储过程。
  3. 当传送数据过快，队列长度大于 2 秒数据量时，不是删除队列头部的整块冗余数据，而是抽样地、等间隔地删除队列中的多个数据，避免有效信息因为传输过快而丢失。

第 3 条引发的问题是，定时器的时间间隔为多长合适？从队列中取数是否需要额外的线程？是否需要加锁？笔者的考虑是：

  1. Python 的 Timer 定时器是多线程的操作，创建线程需要开销，因此定时精度有限，100ms 已看出明显的误差。1s 是比较精确的，减少存数的频次也有利于软件的流畅性，因此使用 1s 的 Timer。
  2. WiFi 数据传送的频次较低，每秒 10-20 条数据包，如果定时器的数据存储操作时间花销较小，可以不考虑锁。并且队列维持了比 1 秒多的数据列，因此存储过程不会出现 index 溢出的现象。所以最后没有使用锁。

流畅显示的线程

波形界面的显示由 matplotlib 负责，图表重绘的花销比数据存储要多得多，因此如果和队列取数共用同一个定时器，会更加影响定时器的精度（精度相关的定时器结构后面说明）。因此我们选择为图表重绘另外提供一个定时器（定时器本身是一个线程）。
需要考虑的问题是，定时器时长多少合适？显示队列头部（早）还是尾部（晚）的数据？1 秒钟绘制多少个点？队列的数据量不够 1 秒怎么办？超过 1 秒怎么办？绘制的时间花销较大，会和网络接口更新队列冲突吗？冲突的话怎么办？

  1. 定时器定时时长选用 1 秒，每秒在图表更新最新 1 秒的数据。因为重绘需要几十毫秒，因此定时越短，精度越低。
  2. 取队列尾部最新的数据显示，保证实时性显示的要求。而实际的数据存储会慢 1 拍。
  3. 队列的数据量少于 1 秒，则从剩余数据中等间隔取 10 个点显示。如果数据量多于 1 秒，则取最近的 1 秒数据量，等间隔抽取 10 个点显示。
  4. 绘制的时间花销较大，会和网络接口更新队列冲突，表现为数组操作中出现偶尔 index 溢出的情况。对应的冲突是，原本队列有超过 1 秒的数据量，但是由于另一个线程更新队列，删除了 1 秒的数据，于是队列变短，index 溢出。解决方案是，在绘图线程启动时，及时一次性从队列中取出需要的数据，实现隔离操作。

容易出错的问题

Python 字符串拼接的效率

本项目中，数据包经解析后最终需要以字符串的形式存储在 .txt 文档里。由于数据写入硬盘是在最后进行，写入之前数据存储在内存中。在最初笔者以字符串变量存储这部分数据，最后变量的大小可能会达到上百兆。但真正的问题是，Python 对字符串的“更新”比想象的要慢很多。

在 Python 中，字符串(string)对象是不可变的(每次关联一个新的字符串变量都会在内存中创建一个新的对象)(译注:类同于 Java,.NET 等现代语言，他们都会在其VM中保留一个字符串池，里面保存所有产生的目标字符串值或临时字符串值)。这方面它与perl、VB等语言中的字符串变量可以任意修改有所不同。如果使用一些比较显而易见的方法(比如：每次都是在新产生的字符串末尾添加一个新短字符串片段)从一些短字符串片段构造长字符串在 Python 中可能会不是很有效率。每次你的在字符串末尾添加内容，Python 解释器都会创建一个新的对象并且复制新产生的对象和原来的对象到解释器中(译注:应该是复制到 Python 解释器的字符串常量池中)。随着处理的字符串的增多，这样的处理过程将会越来越慢。

    时长是指一个视频的时间长度。一个好的样本光是满足前 4 条还不够，必须有较长的时长才能提供丰富的台标样本。

台标提取方法与结果对比

台标样本由视频样本中挑选有限数量的帧产生，为了在单个的视频中尽可能产生互不相同的样本，需要对视频流中相邻前后两个样本做最大相似度限定，使得取后一个样本时，大于相似度限定值的帧被过滤，符合条件的帧成为新的样本。

对于两个帧图像的相似度有不同的量化标准，现有直方图量化法与逐像素取差量化法。前者的计算方法为，计算所有图像的 H-S 直方图，并归一化以便对比。后者的为，统计两张图相似像素的占比。

为了对比两种方法的差异进行了实验。直方图量化法采用 compareHist 函数（参数为 CV_COMP_CORREL），相似度阈值设置为0.05，当为1时表示完全相同。逐像素量化法的占比阈值设置为 70%，当 100% 时几乎完全相同。实验结果如图下图 1 图 2 所示。

OpenCV 实现视频静态台标的提取

发表于 2017-06-13 | 分类于 视频处理

背景介绍

电视台台标在中国出现在电视界面的左上角，标志着电视节目的版权信息。数据统计与监管部门可以通过电视台标统计收视率，监管电视节目的内容。过去的监管需要大量的人力，现在深度学习的出现可以通过台标的识别，减少人力成本。而台标提取是台标识别的前提条件。
根据台标本身是否存在动画，可以分为动态台标与静态台标，本文主要针对静态台标进行提取。而静态台标又可以分为不透明台标与半透明台标。静态台标具有时空不变性，包括位置不变，轮廓不变，颜色不变。利用这种特性，不透明台标可以用基于帧间像素差分的检测方法提取。但是对于透明台标，台标颜色随背景颜色变化，帧间像素差分法不适用。
本文采用了基于边缘检测的提取方法。透明台标虽然颜色会随着背景而变化，但是边缘轮廓会保持不变。另外，台标的识别不仅需要边缘信息，还需要颜色信息等。因此，本文的目标是在边缘算法的基础上提取出完整的台标原图。

操作系统 Win10 64bit，IDE 软件采用 VS 2012，视觉库版本是 OpenCV 2.4.12。

本文采用的方法及流程

  1. Canny 边缘检测并进行时间加权
    Canny 边缘检测算法内置于 OpenCV 中，使用 Canny 边缘检测产生图像的轮廓，白线代表边缘，黑色代表背景。
    为了在视频的多个边缘中筛选出台标的边缘，本文利用台标的时空不变性对 5 帧（每秒取一帧）图像做加权，变动的边缘灰度被减弱，固定的台标边缘灰度保持不变。
    最后经过阈值操作分离出相对干净的台标边缘。
  2. 分水岭操作提取边缘内的台标原图
    分水岭操作的目的是利用边缘图像，生成覆盖台标的 mask，利用 mask 提取原图像帧中的台标。
    分水岭函数需要原图与标识图，而标识图可以由前景与背景合成。因此，本文通过闭运算与膨胀操作生成前景，通过对前景的再膨胀与阈值反色生成背景。
    最后将原图与 mask 做乘法，提取出视频中的台标原图。

Word 的分节符与页眉页脚使用

发表于 2017-05-17 | 分类于 工具

版本说明

Office Word 2016

分节符的分类

分隔符分为分页符与分节符。页是节的组成单元，所以在调整章节之间的关系时，更多的是留意分节符。
分节符分为：

  • 下一页
    在当前页插入分节符，则从下一页开始就是新的一节。
  • 连续
    当前页的分节符后面就是新的一节。使用这种类型的分节符的最常见的原因之一是，如果您正在使用的列。使用此分节符可以更改列的数，而不开始一个新页。
  • 奇数页
    在当前页插入分节符，则从下一个奇数页开始就是新的一节。
    如果当前页是偶数页，则该“奇数页”分节符效果与“下一页”相同；
    如果当前页是奇数页，则下一页在打印成 PDF 时成为空白页（无页眉页脚），但在 Word 软件界面空白页不会显示。从下一个奇数页（也就是“下下页”）开始新的一节。
  • 偶数页
    与奇数页同理，只不过奇数变成了偶数。

不同节的页眉页脚的耦合与去耦合

典型的去耦合场景有，每一节都需要从新的页码“1”开始，首页不显示页码，或者每一节的页眉文字内容不相同。
“耦合/去耦合”的选项出现在双击页眉或页脚的编辑模式下，选项名称为“链接到上一条页眉”。如果选项被选中，则意味着页眉页脚为常量时内容与上一节的一致，为变量时内容默认延续上一节的变量（如页码）。如果选项没有被选中，则页眉页脚在新的一节里重新自定义。

注意事项

不管什么分节场合，强烈不建议在页眉页脚编辑模式下点击“首页不同”。“首页不同”是分节逻辑的快捷按钮，容易造成不分青红皂白地统一或清除页眉页脚格式。因此，即便是需要达成“首页不同”的效果，也从去耦合选项中实现，分节的逻辑更清晰些。
其次，Word 的默认设置是“链接到上一条页眉”，因此每次在新的节中设置页眉页脚，都需要提前注意是否需要去耦合，否则造成后续的编辑混乱。

虚拟机中 Ubuntu16.04 系统 VMTools 的安装

发表于 2017-05-14 | 分类于 工具

遇到的问题

在虚拟机 Ubuntu 系统界面，点击选项安装 VMTools 工具时，出现如下对话框：

正在进行简易安装时，无法手动启动 VMware Tools 安装。

因此，在虚拟机设置选项中选择“软盘选项”，将原来的“使用软盘映像文件”改为“使用物理驱动器”，并加载 VMWare 安装根目录下的 linux.iso 镜像文件，发现软盘里的内容并不能直接出现在桌面，而是在 floppy0 disk 中。
然而，即便以 root 管理者身份登录系统也无法将 floppy0 disk 的压缩包文件到桌面，出现错误提示为：

Error splicing file: Input/output error

软硬件信息

操作系统：i386 32bit 处理器，win7 32bit
使用软件:VMWare Workstation10
虚拟机镜像:Ubuntu 16.04

解决方法

不从软盘上加载镜像，转为直接从一个光盘中“使用物理驱动器”加载，而把剩余的光盘与软盘设置为自动检测。

结果是系统能直接将镜像的内容读取到 Ubuntu 的桌面，不需要从外设中复制。使得剩余的步骤快捷方便。
